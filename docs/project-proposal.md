# 基于 RV1126B 的端侧多模态人机交互终端项目方案

## 1. 项目概述

本项目面向校园工位、学习桌面和个人工作台等轻量场景，基于 ELF-RV1126B 开发板构建一个可离线运行的端侧多模态人机交互终端。系统同时采集语音与视觉信息，通过轻量语音识别、工位物品目标检测和本地任务调度，实现“语音输入、视觉理解、系统执行、界面反馈”的闭环交互。

作品不以通用大模型助手为目标，而是聚焦 RV1126B 端侧算力可承载、现场可稳定演示的实用交互能力。系统能够识别工位常见物品，理解固定语音指令，并调用本地浏览器、播放器等系统软件完成学习、娱乐和找物等任务。

## 2. 赛题契合度

本项目对应瑞芯微赛题指南中的“边缘人工智能应用”和“端侧 AI 视觉应用”方向，重点体现以下能力：

- 端侧 AI 推理：使用 RV1126B 内置 NPU 运行轻量视觉模型，完成工位物品识别。
- 多模态交互：语音模态负责指令输入，视觉模态负责环境感知，两者通过任务调度中心协同工作。
- 实时显示：通过本地图形界面展示摄像头画面、检测框、识别结果、语音文本和任务状态。
- 系统控制：通过本地命令或脚本控制浏览器、播放器、音量和预设网页。
- 工程完整性：形成从采集、识别、解析、执行到反馈的完整闭环，而不是单一模型演示。

## 3. 应用场景

项目以“工位助手”为核心场景，同时覆盖学习效率和生活娱乐两个子场景。

学习效率场景：用户可以通过语音打开学习网页、播放课程视频、查询桌面上是否有书或笔记本，并通过“进入学习模式”触发组合任务。

生活娱乐场景：用户可以通过语音播放音乐、播放视频、暂停播放、打开常用网站，也可以询问“桌上有什么”或“帮我找手机”。

这两个场景共享同一套语音识别、视觉检测、任务调度和界面显示模块，因此互补而不冲突。

## 4. 核心功能分层

### 4.1 必做层

必做层用于保证比赛现场能够稳定演示，是项目第一阶段的最小可行版本。

- 摄像头实时采集桌面画面。
- 识别人、书、笔记本、手机、水杯、鼠标等常见工位对象。
- 通过 GUI 显示摄像头画面、检测框、类别、置信度和系统状态。
- 支持固定语音指令识别，例如“打开浏览器”“打开学习网页”“播放音乐”“播放视频”“暂停播放”。
- 支持基础多模态查询，例如“桌上有什么”“帮我找手机”。
- 通过本地系统命令调用浏览器和播放器完成实际动作。

### 4.2 加分层

加分层用于提升产品完整度和交互体验，在必做层稳定后逐步加入。

- 支持“帮我找水杯”“帮我找书”“有人在工位吗”等更多视觉查询。
- 支持“进入学习模式”“进入休闲模式”等模式切换。
- 支持语音播报反馈，例如“已打开学习网页”“在画面左侧发现手机”。
- GUI 显示最近指令历史、执行结果和当前模式。
- 显示检测帧率、CPU 占用、内存占用和 NPU 推理状态等运行指标。

### 4.3 亮点层

亮点层用于答辩展示，体现多模态协同和任务编排能力。

- “我准备学习了”：系统先检测桌面上是否有书或笔记本，再打开学习网页或课程视频。
- “帮我找书并打开课程网站”：语音意图触发视觉找物，检测成功后执行网页打开任务。
- “进入休闲模式”：系统播放音乐或视频，并切换 GUI 状态。
- Whisper-tiny 增强模式：唤醒后录制 2 到 4 秒短语音，使用 Whisper-tiny 做短句识别，再映射到固定意图。
- 连续运行稳定性展示：记录运行时间、检测帧率、响应延迟和资源占用。

## 5. 识别对象范围

第一版视觉识别对象控制在工位常见物品，避免类别过多导致数据采集、训练和部署风险上升。

优先类别：

- person：人
- book：书或教材
- notebook：纸质笔记本
- phone：手机
- cup：水杯
- mouse：鼠标

候选加分类别：

- keyboard：键盘
- earphone_case：耳机盒或小型收纳盒

第一阶段优先保证前 6 类识别稳定，后 2 类根据数据采集和模型效果决定是否加入最终展示。

## 6. 技术路线

### 6.1 硬件平台

- 主控板：ELF-RV1126B
- 主控芯片：RV1126B
- AI 算力：3TOPS@INT8 NPU
- 摄像头：USB 摄像头或 OV13855 MIPI 摄像头
- 显示：7 英寸 MIPI 显示屏或外接显示方案
- 音频：板载音频接口、USB 麦克风或外接麦克风
- 网络：以太网或 Wi-Fi，用于打开网页和后续远程调试

### 6.2 软件平台

- 操作系统：Debian 12
- 主控程序：Python
- 视觉推理：RKNN Toolkit / RKNN Runtime
- 视觉模型：YOLOv5n 作为第一版主模型，YOLOv8n 作为备选
- GUI：PyQt 作为第一版主界面方案，Web GUI 作为远程调试备选
- 播放器：mpv 或系统自带播放器
- 浏览器：Chromium 或系统可用浏览器
- 语音反馈：TTS 工具或预录音频

### 6.3 视觉模块

视觉模块负责摄像头采集、图像预处理、目标检测、结果缓存和界面绘制。第一版模型选择 YOLOv5n，训练工位物品数据集后转换为 RKNN 格式，在 RV1126B NPU 上部署推理。若后续发现 YOLOv5n 转换或精度不满足要求，再评估 YOLOv8n 或其他轻量检测模型。

输出结果包括目标类别、置信度、位置框和时间戳。调度中心可以查询最新一帧或最近若干帧的检测结果，用于回答“桌上有什么”“帮我找手机”等语音问题。

### 6.4 语音模块

语音主流程采用 Vosk 小型中文离线 ASR 模型，并通过规则进行意图映射，避免把项目成败押在高负载模型上。第一版不做持续语音唤醒，采用按键、界面按钮或短时监听触发录音，重点保证响应速度和识别稳定性。

Whisper-tiny 作为增强功能使用，不承担实时持续监听。推荐在按键或界面按钮触发后录制 2 到 4 秒短音频，再进行短句识别，并将文本映射到固定意图。

### 6.5 指令解析与任务调度

系统设置统一的任务调度中心，将语音文本解析为结构化意图，再根据意图调用视觉模块、系统控制模块、GUI 模块和语音反馈模块。

典型意图包括：

- open_browser：打开浏览器
- open_study_site：打开学习网页
- play_music：播放音乐
- play_video：播放视频
- pause_media：暂停播放
- list_objects：查询桌面物品
- find_object：查找指定物品
- enter_study_mode：进入学习模式
- enter_relax_mode：进入休闲模式

### 6.6 系统控制模块

系统控制模块通过 Python 调用本地程序或命令完成操作。浏览器打开预设网页，播放器播放本地音频或视频，音量控制和暂停播放通过系统命令或播放器接口完成。

所有可执行动作需要设置白名单，避免语音识别误触发任意命令。执行结果返回给 GUI 和语音反馈模块。

## 7. 系统架构

```text
摄像头
  -> 图像采集
  -> YOLO 轻量模型
  -> RKNN/NPU 推理
  -> 场景结果缓存
  -> GUI 显示 / 调度中心查询

麦克风
  -> 唤醒或录音触发
  -> 命令词识别 / 离线 ASR
  -> 指令解析
  -> 任务调度中心

任务调度中心
  -> 查询视觉结果
  -> 调用系统软件
  -> 更新 GUI
  -> 语音反馈
  -> 记录日志
```

## 8. 关键交互流程

### 8.1 “桌上有什么”

用户说出“桌上有什么”。语音模块识别指令后，调度中心读取视觉模块最近一段时间内的检测结果，去重并统计类别，GUI 显示物品列表，语音反馈模块播报结果。

### 8.2 “帮我找手机”

用户说出“帮我找手机”。调度中心查询视觉检测结果中的 phone 类别。如果检测到手机，系统在 GUI 中高亮目标框，并根据目标框中心位置反馈“手机在画面左侧/中间/右侧”；如果未检测到，则提示未发现手机。

### 8.3 “我准备学习了”

用户说出“我准备学习了”。系统先检查当前画面中是否存在 book 或 notebook。如果存在，打开学习网页或本地课程视频，并切换 GUI 为学习模式；如果不存在，提示用户桌面上未检测到学习资料，但仍允许用户手动打开学习网页。

### 8.4 “进入休闲模式”

用户说出“进入休闲模式”。系统启动本地音乐或视频播放，GUI 切换为休闲模式，并记录当前执行动作。

## 9. 演示流程

现场演示建议控制在 5 到 8 分钟。

1. 展示设备启动，GUI 显示实时摄像头画面和工位物品检测结果。
2. 通过语音说“桌上有什么”，系统显示并播报当前识别物品。
3. 通过语音说“帮我找手机”，系统高亮手机并反馈位置。
4. 通过语音说“打开学习网页”，系统调用浏览器打开预设网页。
5. 通过语音说“我准备学习了”，系统结合视觉结果进入学习模式。
6. 通过语音说“进入休闲模式”，系统播放音乐或视频。
7. 展示运行指标，包括检测帧率、语音响应延迟、CPU/内存占用和连续运行时间。

- 一个工位物品目标检测模型及 RKNN 部署文件。
- 一套固定语音指令和意图映射规则。
- 一个本地 GUI 展示界面。
- 一组可稳定演示的学习和休闲场景任务。
- 项目代码、设计文档、演示视频和性能测试记录。

## 11. 第一版验收指标

第一版验收指标用于约束项目范围，避免功能扩张影响核心闭环。

- 视觉识别：至少稳定识别 person、book、phone、cup 4 类对象，后续扩展 notebook 和 mouse。
- 多模态查询：支持“桌上有什么”和“帮我找手机”两个语音触发的视觉查询。
- 系统控制：支持打开学习网页、播放本地音乐、播放本地视频、暂停播放 4 类动作。
- 响应反馈：每次指令执行后，GUI 显示识别文本、执行状态和最近一次结果。
- 演示稳定性：完成 5 分钟连续演示流程，无程序崩溃、无必须联网的核心功能。
- 性能记录：至少记录视觉检测帧率、语音识别耗时、CPU 占用和内存占用。

## 12. 主要风险与应对

### 12.1 Whisper-tiny 实时性风险

Whisper-tiny 在 RV1126B 上可能无法满足实时持续监听需求，因此不作为主流程依赖。主流程使用 Vosk 小型中文离线 ASR 和规则意图映射，Whisper-tiny 仅用于短句识别增强演示。

### 12.2 小目标识别风险

耳机盒、鼠标等小目标可能受摄像头角度、光照和遮挡影响较大。第一阶段优先保证人、书、笔记本、手机、水杯、鼠标 6 类，键盘和耳机盒作为加分类别。

### 12.3 系统调用误触发风险

语音识别可能误触发系统操作。系统控制模块采用动作白名单和二次状态确认，所有可执行命令只允许调用预设网页、预设媒体文件和预设系统动作。

### 12.4 多模块并发风险

视觉、语音、GUI 和系统控制并行运行时可能出现资源抢占。系统采用任务调度中心统一管理状态，视觉推理保持固定频率，语音识别采用触发式运行，避免长时间占满 CPU。

## 13. 阶段计划

### 13.1 第一阶段：最小可行闭环

- 跑通摄像头采集和 GUI 显示。
- 使用轻量目标检测模型识别 3 到 4 类核心物品。
- 实现固定语音指令识别。
- 实现打开网页、播放媒体和查询桌面物品。

### 13.2 第二阶段：模型与交互完善

- 扩展到 6 类核心物品。
- 完成 YOLO 到 RKNN 的转换与 NPU 部署。
- 加入“帮我找手机”“我准备学习了”等多模态指令。
- 加入语音反馈和指令历史。

### 13.3 第三阶段：展示与优化

- 加入 Whisper-tiny 短句识别增强模式。
- 优化 GUI 展示效果和演示流程。
- 记录帧率、响应延迟、资源占用和连续运行稳定性。
- 整理开源仓库、项目文档和演示视频。

## 15. 当前结论

本项目的优势在于场景真实、演示成本低、技术路线贴合 RV1126B 特性，并且能够覆盖赛题中对边缘 AI、多模态交互、实时显示和系统控制能力的考察。项目应坚持“轻量模型、稳定闭环、真实演示”的原则，优先完成必做层，再逐步加入加分层和亮点层。
